# DevTrace MVP 1

## 1. Назначение

`DevTrace` — учебный прототип инструмента для первичного анализа трассируемости между требованиями и исходным кодом.

Цель текущего MVP 1:

- загрузить документы проекта;
- извлечь требования;
- извлечь фрагменты Python-кода;
- найти кандидатов на соответствие `требование -> код`;
- сгенерировать черновые test cases;
- показать матрицу верификации.

Документ фиксирует именно актуальный объём MVP 1 для сдачи и демо.

## 2. Целевая аудитория

Основной пользователь:

- верификатор;
- QA-инженер;
- инженер-программист в области, предъявляющей строгие требования к покрытию кода;
- разработчик, которому нужен быстрый черновой анализ трассируемости кода на требования.

## 5. Выходные данные анализа

После запуска анализа система формирует:

1. Список извлечённых требований;
2. Список извлечённых фрагментов кода;
3. Список кандидатов `requirement -> code chunk`;
4. Список черновых test cases (TODO);
5. Матрица верификации.

## 6. Функциональные требования

### FR-001. Создание проекта

Система должна позволять создать проект.

Минимальные поля:

| Поле | Описание |
|---|---|
| name | название проекта |
| description | краткое описание |

### FR-002. Загрузка документа требований

Пользователь должен иметь возможность загрузить файл требований в проект.

Тип документа:

```text
requirements
```

### FR-003. Загрузка документа исходного кода

Пользователь должен иметь возможность загрузить Python-файл исходного кода в проект.

Тип документа:

```text
source_code
```

### FR-004. Извлечение требований

Система должна извлекать требования из текста по регулярному шаблону.

Поддерживаемый шаблон:

```text
REQ-...: текст требования
```

Минимальные поля извлечённого требования:

| Поле | Описание |
|---|---|
| requirement_key | идентификатор требования |
| text | текст требования |

### FR-005. Извлечение фрагментов кода

Система должна извлекать фрагменты Python-кода по функциям:

```python
def function_name(...):
async def function_name(...):
```

Минимальные поля code chunk:

| Поле | Описание |
|---|---|
| name | имя функции |
| content | текст фрагмента |
| start_line | начальная строка |
| end_line | конечная строка |

### FR-006. Поиск candidate links

Система должна находить по одному лучшему кандидату для каждого требования.

Алгоритм текущего MVP 1:

```text
TF-IDF vectorization
cosine similarity
top-1 candidate code chunk
```

Минимальные поля кандидата:

| Поле | Описание |
|---|---|
| requirement_key | ID требования |
| requirement_text | текст требования |
| code_chunk_name | имя найденной функции или `null` |
| code_chunk_content | текст найденного фрагмента или `null` |
| similarity_score | оценка схожести |
| candidate_status | автоматический статус |

Статусы:

| Условие | Статус |
|---|---|
| `score >= 0.20` | `candidate_found` |
| `0.10 <= score < 0.20` | `ambiguous` |
| `score < 0.10` | `no_candidate_found` |

### FR-007. Генерация тест-кейсов (TODO)

Система должна автоматически формировать минимум один черновой test case на каждое извлечённое требование.

Минимальные поля:

| Поле | Описание |
|---|---|
| requirement_key | ID требования |
| external_id | ID test case |
| title | заголовок |
| preconditions | предусловия |
| steps | шаги |
| expected_result | ожидаемый результат |
| test_type | тип теста |
| status | статус |

Принцип формирования ID:

```text
TC-<requirement_key>-001
```

### FR-008. Построение матрицы верификации

Система должна формировать матрицу верификации по требованиям, найденным кандидатам и draft test cases.

Минимальные колонки:

| Колонка | Описание |
|---|---|
| Requirement ID | ID требования |
| Requirement Text | текст требования |
| Candidate Code | кандидат в коде |
| Similarity Score | оценка схожести |
| System Status | `candidate_found / ambiguous / no_candidate_found` |
| Test Cases Count | количество test cases |
| Verifier Status | техническое поле статуса верификатора |
| Verifier Comment | техническое поле комментария |

## 8. Ограничения текущего MVP 1

В текущую версию не входят:

- авторизация;
- ручное изменение статуса верификации и комментирования;
- отдельный редактор тест-кейсов (заглушка);
- ручное добавление тест-кейсов (заглушка);
- Экспорт в CSV;
- Docker / docker-compose;
- OpenAI API, embeddings, vector DB. (предполагается полноценный ИИ-ассистент)
